# 行思 · Xing-Si — 产品需求文档(PRD)

> **版本**:v1.0(网页端 MVP)
> **范围**:仅描述 Web 端。不包含移动 App、桌面客户端、API 开放等。
> **核心链路**:输入随笔 → 可视化探索 → 共鸣交流 → 续写沉淀

---

## 一、产品定位

一款面向**写作者与思考者**的 AI 应用。

用户输入一段随笔后,系统识别其中的主题、情绪与观点,匹配与之共鸣的**作家、名人或新闻评论**,并支持点击任一对象进入角色化对话,帮助用户继续写作。

**核心主张**:不是替用户写完,而是建立一条可感知的「**输入 — 共鸣 — 对话 — 续写**」链路。

### 1.1 与同类产品的差异

| 类目 | 通用 AI 写作助手 | 行思 |
|---|---|---|
| 默认动作 | 帮我写完 | 帮我看清自己写了什么、与谁共鸣 |
| AI 角色 | 单一万能助手 | 双层 LLM,先匹配再扮演 |
| 关键体验 | 文本输入输出 | 思想可视化 + 角色化对话 |
| 用户产出 | 一段成文 | 一次思想穿越的过程记录 |

### 1.2 目标用户

* **写作者**:随笔/专栏/小说作者,需要灵感碰撞与风格参考。
* **思考者**:习惯写日记、做读书笔记、有思辨需求的用户。
* **学生与研究者**:寻找跨领域对照与反方观点的人。

---

## 二、核心场景

整个产品由三个场景串联,网页端各对应一组屏幕。

```
场景 A · 随笔输入        场景 B · 可视化探索         场景 C · 共鸣交流
─────────────────       ─────────────────         ─────────────────
首页 / 写作页    →    思想星图 / 候选清单    →    对话室 / 灵感卡片
   ↑                                                    ↓
   └──────────────── 续写回到写作页 ────────────────────┘
```

## 三、场景 A · 随笔输入

### 3.1 用户故事

> 我刚刚读完一本书,脑子里有些想法在转,但不知道该跟谁聊。我打开行思,把这段念头写下来。

### 3.2 屏幕

#### A1 · 随笔簿(首页)

* **入口**:站点根路径 `/`。
* **内容**:
  * 历史随笔列表(按时间倒序,卡片式)。
  * 顶部「新起一篇」按钮 → 进入 A2。
  * 每张卡片显示首句、字数、写作日期、若已经匹配过则展示前 3 位回响者头像。
* **空状态**:首次访问展示一段引导文字 + 直接的输入框,降低门槛。

#### A2 · 书写页

* **核心区**:大尺寸文本输入框,默认聚焦,placeholder 提示「写下你正在想的事⋯⋯」。
* **辅助区**:
  * 实时字数统计。
  * 字数 < 20 时,「寻找共鸣」按钮置灰并提示「再多写一点,让回响更准」。
  * 自动保存草稿(本地存储 + 后端持久化,每 5 秒一次或失焦时)。
* **主操作**:`寻找共鸣` 按钮 → 触发 LLM1,跳转 A3。
* **次操作**:
  * 「保存为草稿」(不触发匹配)。
  * 「重新开始」(清空输入,需二次确认)。

### 3.3 输入校验

| 校验项 | 规则 | 提示 |
|---|---|---|
| 最少字数 | ≥ 20 字(中英文统一字符计数) | 「再多写一点,让回响更准」 |
| 最大字数 | ≤ 3000 字 | 「这段思考很长,先截取一段开始吧」 |
| 内容安全 | 提交时做敏感词与违规内容初筛 | 命中则不发起匹配,提示用户修改 |
| 语言 | 中文优先,兼容英文 | 自动识别,无需用户切换 |

### 3.4 数据契约

```ts
type Essay = {
  id: string;
  author_id: string;
  content: string;            // 原文
  word_count: number;
  language: 'zh' | 'en' | 'mixed';
  created_at: ISO8601;
  updated_at: ISO8601;
  status: 'draft' | 'submitted' | 'matched';
}
```

## 四、场景 B · 可视化探索

### 4.1 用户故事

> 我点了「寻找共鸣」之后,屏幕变成一片星图。每颗星是一个曾经写过相似念头的人。我能看出我的想法离鲁迅近一些,离某位评论员远一些——那种空间感本身就是一种回答。

### 4.2 屏幕

#### B1 · 寻回声(匹配中)

* **目的**:为 LLM1 的处理时间提供有温度的等待体验。
* **预计时长**:3–8 秒。
* **内容**:
  * 居中显示用户随笔的首句(或自动提炼的一个短句)。
  * 周围环绕「正在调阅 · 作家库 / 名人库 / 新闻库」三道光晕,依次点亮。
  * 底部进度条 + 「正在为你寻找回响者⋯⋯」文案。
* **失败兜底**:若 8 秒内未返回,降级为加载更长动画;若 30 秒超时,提示用户「线路有些拥挤,稍后再试」并返回 A2,**不丢失原文**。

#### B2 · 思想星图(可视化结果页)

这是场景 B 的核心,呈现 LLM1 的匹配结果。

* **左侧**:用户随笔的精炼摘要 + 自动识别出的 3–5 个标签(主题 / 情绪 / 立场)。
* **中央**:一张二维星图。
  * 中心点 = 用户的随笔。
  * 周围散布 8–12 个回响者节点,**距离表示思想相似度**(越近越像),**颜色表示类型**(作家 / 名人 / 新闻)。
  * 节点大小可表达「该对象在公开语料中相关表达的丰富度」。
* **右侧**:候选清单,与星图节点一一对应。每条卡片显示:
  * 头像 / 标识符号
  * 名称 + 类型徽章
  * 代表摘录(≤ 80 字)
  * 相似度(0–100)
  * **相似点解释**(由 LLM1 生成,1–2 句话说明"为何匹配")。

> ⚠️ 摘录展示遵循版权约束:每条 ≤ 80 字、必带来源标注、不展示完整段落。详见 §8.2。

#### B3 · 视图切换(可选探索)

在 B2 顶部提供视图切换,同一份匹配结果可用四种隐喻浏览:

| 视图 | 隐喻 | 强调维度 |
|---|---|---|
| 思想星图 | 二维散点 | 相似度距离 |
| 时间光谱 | 横向时间轴 | 古今对照 |
| 临水照花 | 上下对照 | 用户文字 vs 回响摘录的逐句映照 |
| 墨年轮 | 同心圆 | 由远到近的影响层 |

> MVP 至少实现 **思想星图** 一种,其余可作为后续迭代。

### 4.3 交互细节

* **悬停节点**:对应右侧卡片高亮 + 显示一行预览摘录。
* **点击节点 / 卡片**:进入场景 C 的对话室(C1)。
* **筛选**:支持按类型(作家 / 名人 / 新闻)、按时代、按相似度阈值过滤。
* **「再找一批」**:重新调用 LLM1,补充第二批候选;原候选不消失,标记为「第一轮」。

### 4.4 LLM1 — 匹配引擎

LLM1 负责理解与匹配,完成三件事:

1. **解析输入文本**的主题、情绪、立场与表达风格,产出结构化标签。
2. 在**作家摘录库、名人观点库、新闻评论库**中检索候选(向量检索 + 关键词召回)。
3. 对候选**重排**,并为每条结果生成「相似点解释」。

#### 4.4.1 输入

```ts
type LLM1Input = {
  essay_text: string;
  essay_id: string;
  user_locale: 'zh' | 'en';
  top_n: number;               // 默认 10
  filters?: {
    types?: ('writer' | 'celebrity' | 'news')[];
    era_from?: number;
    era_to?: number;
  }
}
```

#### 4.4.2 输出

```ts
type LLM1Output = {
  essay_summary: string;                 // ≤ 60 字
  tags: { theme: string[]; emotion: string[]; stance: string[] };
  candidates: MatchCandidate[];
}

type MatchCandidate = {
  id: string;
  name: string;
  type: 'writer' | 'celebrity' | 'news';
  era?: string;                          // 例如 "1881–1936"
  excerpt: string;                       // ≤ 80 字
  excerpt_source: string;                // 出处,例如 "《野草》"
  similarity: number;                    // 0–1
  similarity_reason: string;             // 1–2 句说明
  position_hint: { x: number; y: number }; // 用于星图布局,LLM1 不必给精确坐标,可由前端基于向量降维计算
}
```

#### 4.4.3 质量与安全约束

* **真实性**:仅引用公开发表的作品或言论;不得编造名人引语。
* **版权**:摘录长度受限(§8.2);若候选库未收录可用摘录,该候选不出现在结果中。
* **平衡性**:同一立场的匹配结果不超过 70%,避免回音壁;保留至少 1–2 条**异见或反方**视角。
* **去重**:相同作者的不同摘录,每批至多保留 1 条。
* **超时**:LLM1 单次调用 ≤ 8 秒,超时则返回部分结果 + 标记。

## 五、场景 C · 共鸣交流

### 5.1 用户故事

> 我点了一位看起来"距离我最近"的回响者。屏幕变成一封信纸的样子,他在用他特有的语气回应我刚刚的那段念头。我可以让他继续写下去,也可以让他反驳我。最后我把对话里最打动我的一句话存了下来,贴回我的随笔旁边。

### 5.2 屏幕

#### C1 · 对话室

* **顶部**:回响者标识区
  * 头像 / 名字 / 类型徽章 / 简短介绍(≤ 40 字)。
  * 「这不是真实的 [作者名]。这是基于公开作品的风格化模拟」**永久可见**的免责说明。
* **左侧**(可折叠):用户原始随笔,作为对话上下文锚点。
* **中央**:对话流。
  * 进入时 LLM2 自动生成第一段回应(基于 LLM1 给出的 `similarity_reason` 与摘录上下文)。
  * 用户消息与回响者消息左右分列,字体与语气随回响者风格变化(详见 §5.4)。
* **底部**:输入框 + 五个**对话动作按钮**:

| 动作 | 描述 |
|---|---|
| 让 ta 回应这段话 | 默认动作,针对用户最新发言 |
| 让 ta 续写 | 顺着用户原随笔往下写一段 |
| 让 ta 提出反驳 | 从该角色的立场质疑用户 |
| 从相反立场重审 | 暂时跳出角色,扮演对立观点 |
| 给出更深一层的角度或案例 | 引入新案例或更抽象的视角 |

* **右侧**(可折叠):**灵感卡片**抽屉。
  * 用户可对任意一条回响者发言点击「收藏」→ 自动生成一张灵感卡片,带头像、出处时间、原话片段。
  * 卡片可在场景结束时整体导出,或贴回随笔(见 §5.5)。

#### C2 · 灵感卡片(独立页面)

* 入口:对话室右上角「我的卡片」、或 A1 首页侧栏。
* 内容:所有从对话中沉淀的卡片,按时间或所属随笔分组。
* 操作:
  * 复制卡片文字。
  * 导出为图片(分享场景)。
  * 一键回到原对话上下文。
  * 删除。

### 5.3 LLM2 — 角色对话引擎

LLM2 基于该对象公开作品和观点的思想特征生成风格化回应,模拟真人发言。

#### 5.3.1 输入

```ts
type LLM2Input = {
  persona: {
    candidate_id: string;
    name: string;
    type: 'writer' | 'celebrity' | 'news';
    style_profile: StyleProfile;        // 由 LLM1 阶段预生成或离线构建
    knowledge_snippets: string[];       // 该对象的公开摘录,作为参考
  };
  essay_context: string;                // 用户原随笔
  dialogue_history: Message[];          // 多轮历史
  action: 'respond' | 'continue' | 'rebut' | 'opposite_view' | 'deepen';
  user_message?: string;
}

type StyleProfile = {
  tone: string[];                       // 例如 ["克制", "讽刺", "凝练"]
  sentence_length: 'short' | 'medium' | 'long';
  rhetoric: string[];                   // 例如 ["反问", "白描"]
  vocabulary_register: 'classical' | 'modern' | 'colloquial';
  recurring_motifs: string[];           // 例如 ["铁屋", "看客"]
}
```

#### 5.3.2 输出

```ts
type LLM2Output = {
  message_id: string;
  text: string;
  cited_snippets?: { text: string; source: string }[];  // 引用其公开摘录,≤ 80 字
  suggested_followups?: string[];                       // 3 条建议追问
}
```

#### 5.3.3 角色化要求

* **风格优先于内容**:回应必须明显带有该对象的语言特征(句式、用词偏好、惯用意象)。
* **不得伪造事实**:不得让角色"承认""透露""声称"任何未在公开材料中说过的话。需要发表立场时,基于其已公开立场延展,而非新造。
* **拒绝越界**:用户要求该角色谈论其完全没涉及过的领域时,角色应礼貌承认局限——「这超出我熟悉的话题」。
* **新闻类回响者**:以**评论文章的口吻**回应,不可扮演事件当事人。
* **持续性提醒**:每 10 轮对话或用户主动询问"你是真的吗"时,系统自动插入一句"我只是基于公开作品的模拟"提示。

### 5.4 视觉与语气适配

| 回响者类型 | 信纸样式 | 字体 | 头像 |
|---|---|---|---|
| 作家 | 古纸 / 竖排可选 | 衬线(宋体系) | 印章或剪影 |
| 名人 | 现代信笺 | 非衬线 | 头像 |
| 新闻评论 | 报纸排版 | 报刊衬线 | 媒体 logo + 评论员名 |

所有视觉变量基于主站点的 token(详见 DESIGN.md §4)。

### 5.5 回流到写作

对话室提供「**带回写作**」操作:
* 选中对话中的某段(自己或回响者的话)→ 「插入到我的随笔」。
* 跳回 A2 书写页,被选段落以**引用块**形式插入光标处,标注来源。
* 继续编辑,可再次发起「寻找共鸣」——形成迭代闭环。

## 六、续写沉淀(贯穿)

虽然不是独立场景,但**写作辅助**是产品价值闭环的关键。它作为 A2 书写页与 C1 对话室的浮层菜单,随时可调出:

* **扩写**为散文、评论或短文(指定目标字数)。
* **改写风格**:克制 / 锋利 / 思辨 / 抒情等。
* **提炼**主题句、结构大纲、小标题。
* **补充**现实案例或反方观点。
* **提炼**可分享的**金句**(用于卡片或社交分享)。

每次辅助生成的内容**不直接替换用户原文**,而是以候选项形式出现在侧栏,由用户决定是否采纳。

## 八、非功能性要求

### 8.1 性能与可用性

| 指标 | 目标 |
|---|---|
| A2 输入响应延迟 | < 50ms |
| LLM1 匹配 P50 / P95 | 4s / 8s |
| LLM2 单条回应 P50 / P95 | 3s / 7s,支持流式输出 |
| 草稿自动保存间隔 | 5s 或失焦时 |
| 移动浏览器兼容 | iOS Safari 15+ / Chrome Android 100+(主流程可读) |
| 桌面浏览器 | Chrome / Safari / Firefox / Edge 最近两个版本 |

### 8.2 内容合规与版权

* **摘录长度**:每条 ≤ 80 字,必须附出处。
* **引用频次**:同一作者在同一会话内出现的摘录不重复;在线下语料库构建时遵守合理使用原则。
* **新闻评论**:仅引用公开发表的署名评论;不引用纯事实报道,不重写新闻原文。
* **真伪标识**:对话室任何时刻都可见「这不是真实的 [作者名]」声明。
* **未成年人保护**:候选库不包含针对未成年人的真人模拟;若用户输入指向未成年人,触发安全策略并提示。
* **政治与争议话题**:LLM1 在匹配时保持多元立场;LLM2 在生成时不为单一立场背书。

### 8.3 隐私

* 用户随笔默认**仅自己可见**;无公开分享功能(MVP 不做社区)。
* 用户可随时删除某篇随笔与对应的全部对话、卡片。
* 用于改进模型的数据使用须**独立明确授权**,默认关闭。

### 8.4 可访问性

* 文本对比度满足 WCAG AA。
* 主要交互可键盘完成(Tab / Enter / Esc)。
* 星图视图为视觉强依赖,需提供**列表替代视图**(右侧候选清单本身已具备此能力)。
* 主题面板的「夜读」模式默认对眩光敏感用户友好。

## 九、数据与状态

### 9.1 主要实体

```
Essay         随笔
  └─ MatchRun  一次匹配运行
       └─ Candidate  候选回响者(冗余存储 LLM1 当时的摘要)
            └─ Dialogue  对话会话
                 └─ Message  对话消息
                      └─ Card  收藏的灵感卡片
```

### 9.2 状态机(随笔)

```
draft ──[寻找共鸣]──▶ matching ──[LLM1 完成]──▶ matched
                          │                        │
                          └──[失败]──▶ draft       └──[新一轮]──▶ matching
```

## 十、MVP 范围与后续迭代

### 10.1 MVP 必须包含

* ✅ 场景 A 全部(A1 + A2)
* ✅ 场景 B 的 B1 + B2,且 B2 至少实现**思想星图**视图
* ✅ 场景 C 的 C1 + C2,且 C1 提供五个对话动作中**至少前三个**(回应 / 续写 / 反驳)

## 十一、关键风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| LLM2 让角色"说"未说过的话 | 法律 / 信任风险 | 风格化但不伪造立场;每次回应可被引用至公开摘录;免责声明永久可见 |
| LLM1 形成回音壁 | 思想多样性下降 | 强制保留 ≥ 1 条异见;同立场比例 ≤ 70% |
| 用户依赖 AI 代写 | 偏离产品主张 | 辅助产物以候选形式呈现,不替换原文;UI 引导用户"用自己的话改写" |
| 匹配候选库覆盖不足 | 体验冷启动差 | 优先构建中文核心作家库(50+ 位)+ 主流新闻评论库;允许「再找一批」补充 |
| 摘录版权纠纷 | 法律风险 | 严格遵守 80 字以内 + 必带出处;建立投诉下架通道 |
